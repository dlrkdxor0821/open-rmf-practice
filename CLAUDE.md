## What this repo is

A **study / learning** repo (목적: 공부, not implementation or deployment) for the **Open-RMF** portion of **ABA** (Arte Book Assistance — a library book delivery/retrieval mobile manipulator). It mirrors the ABA monorepo layout but only the `fleet/src/libi_rmf_*` packages are actually worked on; everything else is placeholder scaffolding for context.

Docs, commit discussion, and the author's notes are in **Korean** — match that when writing docs/comments here. The two source-of-truth documents are:
- `docs/study-plan.md` — the learning ladder (M1~M5), the nav2-vs-RMF distinction, and the known traps.
- `docs/aba-architecture.md` — original ABA architecture notes (its §3 folder tree is the *pre-restructure* layout; the real layout is in `README.md`).

## Core mental model: nav2 ≠ RMF

This is the single most important concept and the author's main learning goal:
- **nav2** = how *one* robot drives A→B (path planning, obstacle avoidance, costmaps). The robot's *driver*.
- **RMF** = commanding *many* robots (who does what, intersection yielding, shared doors/lifts). The *control tower*. RMF does **not** do obstacle-avoidance path planning — it only does traffic management on a pre-drawn **nav graph**.

Consequence: the M1~M4 ladder runs RMF with a **slotcar** robot model and **no nav2 at all**. nav2 only appears in **M5** (real-robot integration via EasyFullControl + domain_bridge). "RMF = nav2 + params" is wrong.

## Build & run

ROS 2 **Jazzy**, Gazebo **Harmonic** (`gz sim`). Note: **nav2 is not installed** (needed only at M5) and **rmf_demos is not installed** (needs a source build on the jazzy branch — see `~/open-rmf-test/README`).

```bash
# 1) base ROS
source /opt/ros/jazzy/setup.bash
# 2) deps — REQUIRED before build, or xacro etc. are missing and the build fails (run from the repo root)
rosdep install --from-paths . --ignore-src -r -y       # first time ever: sudo rosdep init && rosdep update
# 3) build everything from the ROOT
colcon build --symlink-install
# 4) apply
source install/setup.bash
```

- **Always build from the repo root during study.** `controller/libi-drive-controller/`, `controller/libi-handy-controller/`, and `fleet/` are three *independent* colcon workspaces (split for per-board hardware deployment), but mixing root-build and per-ws-build scatters the `install/` space — pick root and stay there.
- **Sim-only subset (M1~M4)** to avoid pulling real-hardware deps:
  ```bash
  rosdep install --from-paths controller/libi-drive-controller/src/pinky_pro/pinky_description \
                              controller/libi-drive-controller/src/pinky_pro/pinky_gz_sim --ignore-src -r -y
  colcon build --symlink-install --packages-select pinky_description pinky_gz_sim
  ```
  To stop rosdep dragging in heavy nav2 deps: add `--skip-keys "navigation2 nav2_bringup"`.
- `--symlink-install` lets launch/config/python edits apply **without a rebuild**.

## M1 map → navgraph workflow (`scripts/rmf/`)

The M1 trap: the Gazebo **world** is generated by `pgm_to_gazebo` (from the `.pgm`) while the **navgraph** is generated from a traffic_editor `building.yaml` — two different sources, so their coordinates drift. The fix is a corresponding-points 2D similarity transform (scale·rotation·translation). M1 is **done** (RMS ≈ 1cm).

```bash
# 1) generate navgraph + print its (src) coords
scripts/rmf/make_navgraph.sh fleet/src/libi_rmf_maps/maps/library/new_map.building.yaml
# 2) show the map in rviz at real coords to read dst points (click 'Publish Point', then the map)
ros2 launch scripts/rmf/view_map.launch.py map:=fleet/src/libi_rmf_maps/maps/library/new_map.png.yaml
ros2 topic echo /clicked_point        # point.x/.y = dst
# 3) write new_map.pairs.yaml (copy pairs.example.yaml): src = step-1 output, dst = step-2 clicks, 2+ far-apart pairs
# 4) run the correction (output: /tmp/out/0_fixed.yaml — small RMS residual = success)
scripts/rmf/make_navgraph.sh \
  fleet/src/libi_rmf_maps/maps/library/new_map.building.yaml \
  fleet/src/libi_rmf_maps/maps/library/new_map.pairs.yaml
```

- `fix_navgraph.py` is the correction core (stdlib only, no numpy). `show_navgraph.py` is a standalone rviz-marker node for nav graph lanes/vertices, reusable later inside the Gazebo-test rviz.
- The corrected navgraph is a **derived artifact — not committed**. It is regenerated from source (`building.yaml` + `pairs.yaml`); at M2 it will be generated during build/launch into `install/`.
- Map files live in `fleet/src/libi_rmf_maps/maps/library/` (`new_map.{pgm,png,yaml,building.yaml,pairs.yaml}` + `new_map.png.yaml`). The Gazebo `.sdf` lives in `pinky_gz_sim/worlds/`. Keep `new_map.png.yaml` and `new_map.yaml` on the **same origin/resolution** or dst coords drift.

## Repository structure

Monorepo follows the **pingdergarten** convention: top-level functional folders; **ROS 2 code only** inside colcon workspaces (`controller/*/src/`, `fleet/src/`); non-ROS code (`service/ app/ web/ db/`) lives outside ROS.

```
controller/            # [Equipment] on-robot ROS2 — one colcon ws per board
  libi-drive-controller/src/{libi_drive, pinky_pro}   # nav2 driving (Drive Board)
  libi-handy-controller/src/libi_handy                # MoveIt arm (Handy Board)
fleet/src/             # ★ RMF fleet management — the study focus (its own colcon ws)
  libi_rmf_maps/         building.yaml → world/navgraph        (ament_cmake)   M1 ✓
  libi_rmf_fleet_adapter/ RMF↔robot translation ★ core         (ament_cmake/C++) M2→M5
  libi_rmf_bridge/       domain_bridge config                  (ament_cmake)   M5 only
  libi_rmf_tasks/        tasks + arm perform_action hooks       (ament_python)  M4
  libi_rmf_bringup/      launch assembly                        (ament_cmake)   M2~
service/ app/ web/ db/  # non-ROS backend/clients — placeholders only
scripts/rmf/  docs/  pyproject.toml  docker-compose.yml
```

The `libi_rmf_*` packages mostly contain only `README.md` + `.gitkeep` so far — each README states the package's build type, its `rmf_demos` counterpart, and its role/milestone. **`libi_rmf_fleet_adapter` is the core learning target** (C++ EasyFullControl is the canonical form; read the Python `rmf_demos_fleet_adapter` first, then port).

### Learning ladder (M1~M5)
M1 map (pgm→world + traffic_editor navgraph, coord alignment) → M2 single pinky **slotcar** → M3 multi-robot + traffic negotiation → M4 delivery task + arm `perform_action` → M5 real nav2 + EasyFullControl + domain_bridge. The only big M2→M5 change is swapping the fleet adapter's robot-API backend from *slotcar topic* to *nav2 NavigateToPose*.

## Conventions & gotchas

- **Don't restructure the scaffold without asking.** The layout was deliberately migrated from the ABA-notes form (`src/` umbrella) to the pingdergarten form. Confirm with the user before changing structure.
- **One package = one build type** (ament_cmake = C++, ament_python = Python). Packages are `libi_*`; the robot model is the vendored `pinky_pro` URDF.
- **Two separate Python dependency pools — do not mix them.** ROS nodes use *system* python3 (`package.xml` + rosdep + colcon). Non-ROS `service/app/web/scripts` deps are declared in the root `pyproject.toml` extras (`pip install -e ".[ai,web,dev]"`; `dev` = pytest, ruff). No tests exist yet (`tests/` is a placeholder).
- **`controller/.../pinky_pro` is a fork, not read-only vendor.** It was copied from `github.com/pinklab-art/pinky_pro` (Apache-2.0, keep `LICENSE`) and may be edited directly. But **our project data (maps) stays in `fleet/src/libi_rmf_maps`**, not inside pinky_pro — keep code edits and map data separate.
- `controller/` and `fleet/` are separate workspaces; for nav2 (M5) to reference a `libi_rmf_maps` map, **source both workspaces (overlay)**.
