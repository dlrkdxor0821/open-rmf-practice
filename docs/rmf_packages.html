Open-RMF 패키지 역할 정리 — rmf_demos vs libi

ABA(도서관 책 배달 매니퓰레이터) Open-RMF 학습 repo · 2026-06-23

1. rmf_demos 패키지 (기본 레퍼런스 7개)

패키지빌드타입역할 (상세)
rmf_demosament_cmake 데모 전체를 묶는 launch 진입점. common.launch.xml 하나가 RMF 코어를 통째로 켠다 — rmf_traffic_schedule(일정DB)·rmf_traffic_blockade(좁은구간중재)·building_map_server(/map)·rmf_task_dispatcher(경매)· visualization(rviz)·door/lift supervisor. 그 위에 office.launch·hotel.launch 등 시나리오별 launch + tinyRobot_config.yaml 같은 fleet config 보관.
rmf_demos_assetsament_cmake Gazebo 3D 모델·미디어. TinyRobot·DeliveryRobot 모델 SDF(slotcar 플러그인 장착)·메시, dispenser/ingestor(물건 주고받는 워크셀), 환경 오브젝트. RMF 데모가 gz에 띄우는 시각/물리 자산 전부.
rmf_demos_bridgesament_python RMF ↔ 외부 통신 스택 브리지. fleet_socketio_bridge(웹 대시보드 ↔ RMF, socket.io), fleet_robotmanager_mqtt_bridge(MQTT 로봇매니저 ↔ RMF). RMF 토픽을 웹/IoT 프로토콜로 중계. ※ domain_bridge 아님!
rmf_demos_fleet_adapterament_python EasyFullControl 어댑터(RMF↔로봇 통역). fleet_adapter.py(RMF 등록·navigate/execute_action 콜백)· fleet_manager.py(slotcar 백엔드, PathRequest 발행/robot_state 수신)· RobotClientAPI.py(HTTP 다리). config+navgraph 읽어 add_easy_fleet.
rmf_demos_gzament_cmake Gazebo(gz) 시뮬 launch. gz world 띄우고 로봇 spawn, ros_gz bridge 연결. 코어(rmf_demos)와 sim(gz)을 결합하는 시나리오 launch(office.launch.xml 등).
rmf_demos_mapsament_cmake 데모 맵. traffic_editor building.yaml + floorplan 이미지. 빌드 시 building_map_generator로 world + navgraph(0.yaml) 산출.
rmf_demos_tasksament_python task 요청 CLI. dispatch_patrol·loop·delivery·clean·action·go_to_place + cancel. task JSON을 조립해 ApiRequest로 task_api_requests에 발행(submitter).

2. 우리 패키지 (libi_rmf_* · 5개)

패키지빌드타입역할 (상세) rmf_demos 대응
libi_rmf_mapsament_cmake 우리 맵 new_map(library). building.yaml→navgraph + M1 좌표보정(pgm월드↔traffic_editor navgraph 대응점 닮음변환). charger·point_a/b waypoint 명명. rmf_demos_maps ✓
libi_rmf_fleet_adapterament_python EasyFullControl 직역 포팅. 4파일 그대로 + 패키지명/config만 libi로. slotcar 백엔드(M2~4) → M5에 nav2로 백엔드만 교체. execute_action에 pickup/dropoff 팔 동작(M4) 추가. rmf_demos_fleet_adapter ✓
libi_rmf_tasksament_python dispatch_delivery.py — 배달 compose task 조립·제출 (go(shelf)→pickup→go(desk)→dropoff). 우리만의 task 정의가 사는 곳. rmf_demos_tasks ✓
libi_rmf_bringupament_cmake RMF 두뇌 조립 launch. libi_rmf.launch.xml = 코어(schedule·blockade·building_map_server·dispatcher) + 어댑터 include. scripts/rmf.sh가 띄움. M5엔 nav2·domain_bridge까지 여기 합쳐짐. rmf_demos(common.launch) ✓
libi_rmf_bridgeament_cmake domain_bridge 설정(M5). 실로봇이 각자 ROS 도메인에서 nav2 풀스택을 돌릴 때, RMF에 필요한 토픽만 선택적으로 도메인 간 브리지. M5에서만 등장. rmf_demos_bridges 아님 ⚠️
⚠️ 흔한 오해 — libi_rmf_bridge ≠ rmf_demos_bridges:
· rmf_demos_bridges = RMF ↔ 웹/MQTT(socketio·mqtt) 외부 통신 브리지
· libi_rmf_bridge = domain_bridge(실로봇 각자 ROS 도메인 ↔ RMF, M5) — upstream의 별도 도구. rmf_demos가 안 가진 거라 우리가 새로 둔 것. 둘 다 "브리지"지만 목적이 완전히 다름.

3. 빠진 패키지 + 왜 합리적인 선택인가

빠진 rmf_demos 패키지그 역할은 어디서 채우나 왜 빼는 게 합리적
rmf_demos_assets
(로봇/물체 3D 모델)		 controller/pinky_pro/pinky_description (pinky URDF) + slotcar 플러그인은 apt 설치(libslotcar.so) 우리 로봇은 데모 TinyRobot이 아니라 pinky(실 타깃). 로봇 자산은 로봇과 함께 controller/에 둔다(pingdergarten: 장비=controller, 관제=fleet). 데모용 자산을 따로 둘 이유 없음.
rmf_demos_gz
(Gazebo 시뮬 launch)		 controller/pinky_pro/pinky_gz_sim + scripts/libi_sim.sh(tmux 래퍼, MODE 토글) 시뮬은 로봇에 종속 → pinky_gz_sim(로봇 옆)에. RMF 관제(fleet/)와 분리해 책임이 명확. world는 pgm_to_gazebo(M1)로 생성.
rmf_demos_bridges
(MQTT/socketio 브리지)		 service/ · app/ · web/ (비-ROS, 현재 placeholder 미구현) 이건 외부 웹/IoT 연동 계층 → ABA에선 비-ROS 앱이 담당(ROS 바깥). 학습 초점은 RMF 코어라 나중/별도. (domain_bridge와 혼동 주의 — 위 ⚠️)
(rmf_demos 모놀리식 자체)
launch+config 한 덩어리		 libi_rmf_bringup(조립 launch) + config는 libi_rmf_fleet_adapter/config 데모 시나리오 다발(office/hotel…) 대신 우리는 단일 맵 → bringup 하나로 충분. 패키지 책임을 쪼개 더 명확하게.

4. 그대로 인용(포팅)한 것 vs 우리가 직접 구축한 것

open-rmf(rmf_demos) 코드를 어디까지 가져왔고, 어디부터 우리 고유 작업인지.

구분항목출처 · 우리가 바꾼/만든 부분
인용·포팅 fleet_adapter / fleet_manager /
RobotClientAPI		 rmf_demos_fleet_adapter 4파일 직역 포팅(거의 그대로). 바꾼 것: 패키지명·config 경로·상대 import, 그리고 execute_action에 pickup/dropoff 케이스만 추가.
인용·포팅 libi_fleet_config.yaml (구조) rmf_demos tinyRobot_config.yaml 틀 그대로. 값은 우리 것: pinky 속도/footprint, library charger·waypoint, reversible:false, actions:[pickup,dropoff].
인용·포팅 dispatch_delivery.py (골격) rmf_demos_tasks dispatch_action.py의 ApiRequest 발행·compose 조립 패턴 차용. 배달 시퀀스(go→pickup→go→dropoff) 조립은 우리 것.
인용·포팅 slotcar 플러그인 파라미터 TinyRobot slotcar 기본값 차용. 바꾼 것: tire_radius·base_width(pinky 실측), reversible:false(후진금지), 질량.
직접 구축 M1 좌표보정 도구 fix_navgraph.py·make_navgraph.sh·show_navgraph.py — pgm월드↔navgraph 닮음변환(대응점). rmf_demos에 없는 우리 고유 (RMS≈1cm).
직접 구축 library 맵 일습 new_map.{pgm,png,yaml,building.yaml,pairs.yaml} — 우리가 그린 도서관 맵.
직접 구축 실행 스크립트 libi_sim.sh(tmux sim+rviz, MODE 토글)·rmf.sh(bringup 래퍼)·libi_kill.sh(정리).
직접 구축 libi_rmf_bringup launch libi_rmf.launch.xml — 코어+어댑터 조립을 우리가 작성.
직접 구축 pinky-slotcar 통합 pinky_slotcar.urdf.xacro·pinky_gz_slotcar.urdf.xacro — pinky 몸체에 slotcar 붙이는 변형 + diffdrive/slotcar 토글.
직접 구축 M3 2번째 로봇 spawn launch_sim·libi_sim.sh에 조건부 2대 spawn (negotiation 관찰).
직접 구축 RViz TF 연결 fix sim_view.launch.py 정적변환(map→world, base_link→base_footprint) — gz↔rviz 좌표 불일치 해결.
직접 구축 M4 팔 핸들러 perform_arm_action(mock 로그+딜레이 후 finished) — 우리가 추가한 동작 핸들러.
직접 구축 문서 트러블슈팅.md·README·docs/·이 PDF — 학습 기록.
요약: 인용·포팅 = RMF↔로봇 핵심 배선(어댑터 3종) + 설정/요청 골격 — "검증된 표준을 그대로". 직접 구축 = 맵·좌표보정·로봇통합·실행환경·팔동작·문서 — "우리 로봇(pinky)·맵·task에 맞춘 부분". → 바퀴(표준)는 빌려오고, 우리 시나리오에 맞춘 살은 직접 붙였다.
핵심 한 줄
맵·어댑터·tasks·bringup은 rmf_demos를 거의 1:1 미러링, 빠진 3개(assets·gz·bridges)는 역할이 딴 데 있어서 뺀 것:
· 로봇 자산·시뮬(assets/gz) → 로봇과 함께 controller/pinky_pro (우리 로봇은 pinky라 이미 있음)
· 외부 웹/MQTT 브리지(bridges) → 비-ROS service/app/web (앱 계층, 미구현)
· libi_rmf_bridge는 새로 추가 (M5 domain_bridge) — rmf_demos_bridges와는 다른 브리지
→ "RMF 관제(fleet/)에 필요한 것만 미러링 + 로봇·시뮬은 controller/, 외부연동은 app/으로 분리" — pingdergarten 컨벤션을 따른 합리적 선택.
