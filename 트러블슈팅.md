# 트러블슈팅 (Open-RMF / libi — slotcar 시뮬 ~ 실 프로젝트)

이번 M2 작업에서 실제로 겪은 에러들을 **증상 → 원인 → 해결** 순으로 정리. 실 프로젝트에서 같은 증상 나오면 여기서 찾아 대처.

---

## 0. 가장 먼저 (막히면 이것부터)

**① 환경 완전 초기화** — gz/RMF 좀비 프로세스가 충돌의 90%:
```bash
scripts/libi_kill.sh          # 모든 gz/ros/rmf 프로세스 강제 종료 + tmux 정리
scripts/libi_kill.sh list     # 죽이지 말고 떠있는 것만 확인
```

**② 정상 기동 순서** (순서 틀리면 slotcar가 level 못 잡음):
```
1) MODE=slotcar scripts/libi_sim.sh           # sim(gz+slotcar+rviz) — 끝까지 뜰 때까지 대기
2) ros2 launch libi_rmf_bringup libi_rmf.launch.xml \
     building_yaml:=<abs> nav_graph_file:=<abs> use_sim_time:=true   # RMF 코어+어댑터
3) (~15초 대기, /robot_state Publisher 1 확인 후) dispatch
```
> 핵심: **sim 먼저 → bringup(building_map_server) 나중.** 그래야 slotcar가 fresh `/map`을 받아 level을 잡는다.

## 1. 프로세스 / 환경

### 1.1 렉 + 좀비 gz 서버 다수
- **증상**: Gazebo/RViz 극심한 렉, 마커 깜빡임. `ps`에 `gz sim`이 여러 개.
- **원인**: 이전 sim/테스트가 안 죽고 남음. **gz 서버 여러 개가 각자 `/clock`·`/tf`·`/map`을 동시 발행 → 토픽 충돌.**
- **확인**: `scripts/libi_kill.sh list` 또는 `pgrep -fc "gz sim"` (1 초과면 문제).
- **해결**: `scripts/libi_kill.sh`
- **예방**: 테스트/실행 끝낼 때 항상 `libi_sim.sh down` + Ctrl-C, 그래도 남으면 `libi_kill.sh`. headless 백그라운드 실행은 PID로 정확히 kill.

### 1.2 `pkill -f`가 자기 자신을 죽임
- **증상**: 정리 스크립트를 돌리면 **출력 0줄 + exit 1** (아무 일도 안 일어난 듯).
- **원인**: `pkill -f "gz sim"` 의 패턴 문자열이 **그 명령을 실행 중인 셸의 argv에도 들어있어** 자기 자신을 매칭→kill.
- **해결**:
  - 패턴을 **파일 안**에 두고 `bash kill.sh`로 실행 (실행 argv는 `bash kill.sh`뿐 → 패턴 매칭 안 됨). ← `libi_kill.sh`가 이 방식.
  - 또는 백그라운드 PID를 `$!`로 잡아 **PID로 kill**.
- **주의**: `setsid cmd & PID=$!` 의 `$!`는 setsid PID라 **실제 자식 PGID와 다름** → `kill -- -$PID` 가 빗나가 좀비가 남는다. setsid+그룹kill 쓰지 말 것.

### 1.3 `gz` 명령이 PATH에 없음
- **증상**: `gz: command not found`, `which gz` 비어있음 (특히 비로그인/headless 셸).
- **해결**: gz 서버는 `ros2 launch`로 띄운다:
  ```bash
  ros2 launch ros_gz_sim gz_sim.launch.py gz_args:="-r -s -v4 <world.sdf>"   # -s = 서버만(headless)
  ```

### 1.4 `/clock`이 ROS로 안 옴 (headless)
- **증상**: `use_sim_time` 노드가 시간 못 받음. 어댑터/수동 PathRequest의 시간이 sim time과 안 맞아 로봇이 이상하게 움직이거나 안 움직임. `ros2 topic echo /clock --once`가 hang.
- **원인**: gz의 `/clock`은 **gz 토픽**이라 `ros_gz_bridge`로 ROS에 브리지해야 함.
- **해결**: parameter_bridge 실행 (pinky_bridge.yaml에 clock 브리지 포함):
  ```bash
  ros2 run ros_gz_bridge parameter_bridge --ros-args -p config_file:=<pinky_bridge.yaml>
  ```
  - `MODE=slotcar libi_sim.sh`는 이 브리지를 자동 실행. 수동 headless 테스트 땐 직접 띄울 것.
  - 임시방편: PathRequest 시간을 `/robot_state`의 `location.t.sec`에서 뽑아 쓰면 `/clock` 없이도 가능.

## 2. 좌표 / TF / RViz

### 2.1 `/map` 토픽 타입 충돌 → 평면도 안 뜸
- **증상**: RViz에 맵 이미지(평면도)가 안 뜸.
- **원인**: **두 publisher가 같은 `/map`에 다른 타입**:
  - nav2 `map_server` → `/map` (`nav_msgs/OccupancyGrid`, 평면도)
  - RMF `building_map_server` → `/map` (`rmf_building_map_msgs/BuildingMap`, slotcar 레벨용)
- **해결**: `/map`은 slotcar/RMF가 써야 하니 **평면도를 다른 토픽으로 remap**.
  - `sim_view.launch.py`: `map_server` 노드에 `remappings=[("map","floorplan_map")]`
  - `sim_view.rviz`: Map display Topic을 `/floorplan_map`으로
- **확인**: `ros2 topic info /map` → Type이 `BuildingMap` 하나여야 함.

### 2.2 RViz 로봇 위치 ≠ Gazebo / TF 트리 단절
- **증상**: RViz 로봇이 엉뚱한 데(보통 맵 원점 근처)에 멈춰 있음. `ros2 run tf2_ros tf2_echo map base_footprint` → `not part of the same tree. Tf has two or more unconnected trees.`
- **원인**: slotcar 모드엔 `odom→base_footprint` TF가 없다(diff-drive는 DiffDrive 플러그인이 냄). gz는 로봇 pose를 **`world→pinky1/base_link`**(네임스페이스+world 프레임)로만 내보내서 URDF(`base_footprint`)/`map` 체인과 안 이어짐. **Gazebo는 항상 정상 — 순수 RViz(시각화) 문제.**
- **해결 (★검증됨, 권장)**: gz가 **항상 내보내는** `world→pinky1/base_link`(실제 pose)를 정적변환 2개로 연결. `/robot_state`/레벨판정에 **의존 안 함** → 안정적.
  - `sim_view.launch.py` (slotcar_tf=true) 에 추가 (이미 적용됨):
    - `map → world` (identity, 월드≡맵 프레임)
    - `pinky1/base_link → base_footprint` (identity; z 2.8cm 오프셋은 top-down 에 무시)
  - 체인: `map→world→pinky1/base_link(gz)→base_footprint→base_link→바퀴` → RobotModel 이 gz 실제 위치 추종.
  - ※ `parameter_bridge` 가 gz `/tf`(`world→pinky1/base_link`)를 ROS 로 브리지해야 함 (libi_sim slotcar 모드는 자동).
  - ※ 모델명이 pinky1 이 아니면 프레임 `pinky1/base_link` 를 그 이름으로 바꿀 것: `ros2 topic echo /tf --once | grep child_frame_id`.
- **대안**: `/robot_state`(slotcar 위치)로 `map→base_footprint` 직접 발행하는 `robot_state_to_tf.py` — 단 `/robot_state` 가 나와야 동작([3.1](#31--robot_state-publisher-0--level_name-판정-실패-핵심) 선결). **gz pose 방식이 더 안정적이라 그걸 쓴다.**
- **확인**: `ros2 run tf2_ros tf2_echo map base_footprint` → charger 좌표(-3.87, 11.2…) 나오고 Gazebo 와 일치.

### 2.3 TF 시간 점프 (jump back in time)
- **증상**: `TF_OLD_DATA ignoring data from the past`, `Detected jump back in time. Clearing TF buffer.`, RViz가 계속 리셋.
- **원인**: **여러 sim(여러 gz)이 각자 다른 sim-clock으로 `/clock`·`/tf` 동시 발행** (좀비 프로세스). 시간이 두 값 사이를 점프.
- **해결**: `scripts/libi_kill.sh` 로 좀비 정리 → **sim 하나만** 실행. (= [1.1](#11-렉--좀비-gz-서버-다수))

## 3. slotcar / RMF

### 3.1 `/robot_state` Publisher 0 / level_name 판정 실패 (★핵심)
- **증상**:
  - `ros2 topic echo /robot_state --once` 가 **영원히 hang** (메시지 안 옴)
  - `ros2 topic info /robot_state` → **Publisher count: 0**
  - Gazebo 터미널 로그: `[slotcar_xxx]: Unable to determine the current level_name for robot [...]. The RobotState message for this robot will not be published.`
- **파급**: `/robot_state`가 없으면 → RViz 로봇 안 붙음 + RMF가 로봇 online 인식 못 함 + dispatch 무시. **거의 모든 증상의 뿌리.**
- **원인**: slotcar는 **로봇 z-elevation을 building map의 level elevation과 매칭**해 level을 정함(`get_level_name(z)`). `/map`(BuildingMap)을 제때 못 받으면 `_initialized_levels=false` → 판정 실패. 우리 building.yaml은 navgraph용이라 **`floors:` 폴리곤이 없고**, building_map_server가 `unable to generate GeoJSON` 경고를 냄. 게다가 **타이밍/순서에 민감** (latched `/map`을 놓치는 듯).
- **해결 (실전)**:
  1. **building_map_server(bringup)를 slotcar 뒤에 띄운다** → fresh `/map` → slotcar가 재시도 성공. (정상 기동 순서 [0번](#0-가장-먼저-막히면-이것부터))
  2. `ros2 topic info /robot_state` 로 **Publisher 1** 확인. 1 되는 순간 robot online + RViz 로봇 점프.
- **해결 (robust, 타이밍 무관하게)** — 아직 미적용, 실 프로젝트 권장:
  - building.yaml(traffic_editor)에 **`floors:` 폴리곤**을 그려 넣기 (rmf_demos 맵처럼). level 판정이 floorplan/floor 기반으로 안정.
  - 또는 월드 SDF에 RMF level/charger-waypoint 정보 주입 (rmf building_map_generator가 만드는 월드처럼).
- **확인 명령**:
  ```bash
  ros2 topic info /robot_state                 # Publisher count?
  ros2 topic info /map                          # BuildingMap, Publisher 1?
  ros2 node list | grep building_map            # building_map_server 떠있나?
  # Gazebo 터미널에서 "Unable to determine level" 로그 확인
  ```

### 3.2 dispatch는 됐는데 로봇이 안 움직임
- **증상**: `dispatch_go_to_place ... success: True, assigned_to ...` 인데 로봇 정지.
- **원인 후보**:
  - `/robot_state` 미발행 → robot online 아님 → [3.1](#31--robot_state-publisher-0--level_name-판정-실패-핵심).
  - PathRequest 시간이 sim time과 안 맞음 → [1.4](#14-clock이-ros로-안-옴-headless).
  - headless RTF가 낮아 **느리게** 움직이는 것뿐 (mode가 `2`(MOVING)면 움직이는 중). GUI로 보면 정상 속도.
- **확인**: `ros2 topic echo /robot_state --once` → `mode: 2`(MOVING) + `task_id` 채워짐 + `x/y` 변화하면 정상.

### 3.3 dispatch 대상 waypoint 이름이 없음
- **증상**: `dispatch_go_to_place -p <이름>` 인데 그 정점이 navgraph에 없음.
- **해결**: building.yaml 정점에 이름 부여(예: `point_b`) → navgraph 재생성([4.4](#44-navgraph-재생성-buildingyaml-수정-후)). RMF task는 **이름 있는 waypoint**만 목적지로 받는다.

## 4. 빌드 / launch

### 4.1 robot_description "yaml 파싱" 에러
- **증상**: `[launch]: Caught exception ... Unable to parse the value of parameter robot_description as yaml. ... try wrapping it in ... ParameterValue(value, value_type=str)`. RSP가 안 뜸.
- **원인**: xacro 출력(긴 XML)을 robot_description 파라미터로 줄 때 launch가 그걸 YAML로 파싱하려다 실패.
- **해결**: `upload_robot.launch.py` 에서 `Command([...])` 를 `ParameterValue(Command([...]), value_type=str)` 로 감싼다. (적용 완료)

### 4.2 bringup "required arguments"
- **증상**: `ros2 launch libi_rmf_bringup libi_rmf.launch.xml` → `the following arguments are required: building_yaml, nav_graph_file`.
- **원인**: `libi_rmf_maps`가 colcon 패키지가 아니라(맵만 있음) `find-pkg-share`로 못 찾음 → 맵 경로를 **절대경로 인자**로 받게 설계함.
- **해결**: 인자 꼭 주기:
  ```bash
  B=$PWD/fleet/src/libi_rmf_maps/maps/library/new_map.building.yaml
  N=$PWD/fleet/src/libi_rmf_maps/maps/library/new_map.navgraph.yaml
  ros2 launch libi_rmf_bringup libi_rmf.launch.xml building_yaml:=$B nav_graph_file:=$N
  ```

### 4.3 xacro가 신규 파일을 못 찾음
- **증상**: `xacro ... error: No such file ... install/.../share/.../urdf/새파일.xacro`.
- **원인**: `$(find pkg)`는 **install 디렉터리**를 보는데, 새로 만든 파일이 아직 install에 없음 (`--symlink-install`도 신규 파일은 재빌드 필요).
- **해결**: `colcon build --packages-select <pkg> --symlink-install` 후 `source install/setup.bash`.

### 4.4 navgraph 재생성 (building.yaml 수정 후)
- **언제**: building.yaml 정점/이름/charger를 바꾼 뒤.
- **명령**:
  ```bash
  OUT_FILE=fleet/src/libi_rmf_maps/maps/library/new_map.navgraph.yaml \
    scripts/rmf/make_navgraph.sh \
      fleet/src/libi_rmf_maps/maps/library/new_map.building.yaml \
      fleet/src/libi_rmf_maps/maps/library/new_map.pairs.yaml
  ```
  - 좌표 보정(M1)도 같이 적용됨. RMS 잔차 작으면 성공.

## 5. 진단 명령 툴킷

```bash
# 떠있는 RMF/sim 프로세스
scripts/libi_kill.sh list
ros2 node list

# 로봇 상태/위치 (Gazebo 진실)
ros2 topic info /robot_state          # Publisher count (0이면 slotcar 침묵 → 3.1)
ros2 topic echo /robot_state --once   # x/y/yaw/level_name/mode/task_id

# 맵 토픽 (타입 충돌 확인)
ros2 topic info /map                  # BuildingMap 하나여야 (2.1)

# TF (RViz 로봇 위치)
ros2 run tf2_ros tf2_echo map base_footprint   # 끊겨있나/일치하나 (2.2)
ros2 topic echo /tf --once | grep -E "frame_id|child_frame_id"

# 시뮬 시간
ros2 topic echo /clock --once         # 안 오면 1.4

# dispatch (rmf_demos_tasks 소스워크스페이스 source 필요)
ros2 run rmf_demos_tasks dispatch_go_to_place -F libi -R pinky1 -p point_b --use_sim_time
```

## 부록 — 핵심 인과 사슬 (한눈에)

```
building_map_server(bringup) ──/map(BuildingMap)──▶ slotcar 가 level_name 판정
        (없거나 늦으면 → "Unable to determine level_name")              │ 성공해야
                                                                         ▼
                                                          slotcar 가 /robot_state 발행
                                                                         │
                        ┌────────────────────────────────────────────────┼─────────────────────────┐
                        ▼                                                 ▼                         ▼
            robot_state_to_tf → map→base_footprint TF          fleet_manager → 어댑터          RViz 로봇 위치
            (RViz 로봇이 Gazebo와 일치)                         → RMF online → dispatch 가능
```
> **`/robot_state` Publisher 가 1이 되는 것 = 모든 게 살아나는 스위치.** 막히면 항상 여기부터 확인.
